# Security Audit Checklist (Alpha Phase)

This checklist ensures the AI-Automated Auto Insurance Claims Platform meets baseline security requirements before being demonstrated to external clients or handling any sensitive (even if mocked) Personally Identifiable Information (PII).

## 1. Authentication & Authorization

*   [ ] **Admin Portal Access:** Is the admin dashboard protected by strong authentication (e.g., JWT, Clerk)?
*   [ ] **Role-Based Access Control (RBAC):** (If implemented) Are users restricted only to the endpoints and data they are authorized to access?
*   [ ] **API Key Management:** Are external API keys (Stripe, Anthropic, Supabase) securely stored in environment variables and **never** committed to version control?
*   [ ] **Password Policies:** Does the authentication provider enforce strong password requirements (length, complexity)?
*   [ ] **Session Management:** Are JWTs correctly signed, with appropriate expiration times (`exp` claim), and handled securely on the frontend (e.g., HTTP-only cookies or secure local storage)?

## 2. Data Protection & Privacy

*   [ ] **Data in Transit:** Is all communication between the frontend, backend, and database encrypted using TLS/SSL (HTTPS)?
*   [ ] **Data at Rest:** Is the PostgreSQL database encrypted at rest (typically handled by managed providers like Supabase/AWS)?
*   [ ] **PII Handling:** Are fields like `claimant_name`, `phone`, and `email` treated securely?
*   [ ] **Photo Storage:** Are uploaded damage photos stored securely in an S3/R2 bucket with private access?
*   [ ] **Signed URLs:** Are photos only accessible via temporary, signed URLs generated by the backend rather than being publicly readable?
*   [ ] **Database Logging:** Ensure SQLAlchemy `echo` is explicitly set to `False` in production to prevent leaking sensitive data or queries into the application logs.

## 3. Application Security (Backend - FastAPI)

*   [ ] **Input Validation:** Are all incoming API requests (FNOL submissions, admin actions) strictly validated using Pydantic models (V2 `model_config`)?
*   [ ] **Prompt Injection:** Is the input to the Claude/LLM API sanitized to prevent prompt injection attacks (e.g., stripping XML-like tags from user input)?
*   [ ] **Rate Limiting:** Is rate limiting implemented on public endpoints (like `/api/v1/claims` for FNOL submission) to prevent DDoS or spam?
*   [ ] **CORS Settings:** Are Cross-Origin Resource Sharing (CORS) origins strictly limited to the known frontend domains (e.g., `https://claims.your-platform.com`)?
*   [ ] **Error Handling:** Are detailed error traces (stack traces) disabled in production to prevent leaking internal application structures?

## 4. Payment Security (Stripe)

*   [ ] **PCI Compliance:** Does the application avoid directly handling or storing raw credit card/bank account numbers, relying entirely on Stripe Elements or Stripe Connect for sensitive data collection?
*   [ ] **Webhook Verification:** Are incoming Stripe webhooks cryptographically verified using the Stripe Webhook Secret to ensure they actually originated from Stripe?
*   [ ] **Environment Segregation:** Are test Stripe keys (`sk_test_...`) strictly used for the Alpha/Demo phase, with live keys completely separate?

## 5. Infrastructure & Deployment

*   [ ] **Environment Variables:** Are `.env` files explicitly excluded in `.gitignore`?
*   [ ] **Dependency Scanning:** Have backend (`requirements.txt`) and frontend (`package.json`) dependencies been scanned for known vulnerabilities (e.g., using `npm audit` or GitHub Dependabot)?
*   [ ] **Build Artifacts:** Are temporary build/cache files (e.g., Next.js `.next` folder, local `uploads/` directory, `__pycache__`) excluded from version control?
*   [ ] **Least Privilege:** Do the database user and application service accounts run with the minimum necessary privileges?
